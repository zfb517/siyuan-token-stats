# SiYuan Token Statistics

> 思源笔记 Token 用量统计插件 — 统计思源笔记内所有 AI 调用的 Token 消耗（含通过思源智能体代理转发的第三方 API 调用，以及直接调用第三方 API 的请求），支持每个 API Key 单独设置限额、自定义提醒阈值、限额自动切断，以及 S3/WebDAV 云同步数据合并。

## 功能特性

- **按 API Base URL 匹配**：所有 API Key 均按 Base URL 匹配，不再区分"内置"与"外部"。思源 AI 可配置为 `http://127.0.0.1:6806/api/ai/` 或 `/api/ai/`，第三方 API 填写对应 URL
- **思源 conf.json 模型识别**：自动扫描思源工作空间 `conf/conf.json` 配置文件，将请求体中的块 ID（如 `20260705184400-rxdfohi`）解析为真实模型名（如 `deepseek-v4-flash`），精确识别思源智能体实际使用的模型
- **自定义提供商名称**：每个 API Key 的"提供商"字段可任意填写（如 OpenAI、DeepSeek、华为云、SiYuan 等），仅用于显示和图标
- **多级 Key 匹配策略**：思源 AI 请求按 provider.apiKey → 关联模型 → provider.baseURL → provider 模型列表 → URL 兜底顺序匹配；外部 API 按模型 → API Key → URL 匹配
- **关联模型匹配**：同一 Base URL 下配置多个 Key 时，可通过"关联模型"字段按请求模型精确匹配到对应 Key
- **Token 统计**：优先从 API 响应中提取精确的 token 用量，无 usage 字段时自动估算
- **流式支持**：正确处理 SSE 流式响应（含思源智能体自定义事件格式），解析 usage 事件或估算累积文本
- **API Key 管理**：
  - 每个 API Key 可单独设置 Token 限额（0 = 不限）
  - 每个 API Key 可自定义提醒阈值百分比（0-100%）
  - 支持启用/禁用、按 Key 或 URL 匹配
  - 支持导入/导出 Key 配置（JSON 格式）
- **限额自动切断**（三道防线）：
  - 请求前预检：当前用量 + 预估输入超限时拦截
  - 流式中途中断：SSE 生成过程中超限时 `reader.cancel()` 立即中断
  - 受控错误响应：返回 HTTP 429 而非 throw，调用方可正常处理
- **限额重置周期**：支持每月 / 每日 / 自定义（天）/ 永不重置，统计用量时只计算当前周期内的记录；选「自定义（天）」时在紧邻下方的「自定义周期天数（天）」填写天数（1–365），按「最近 N 个自然日」结算（如 14 = 每两周、90 = 每季）；三个重置周期（限额/全局限额/全局费用）共用此天数
- **云同步合并与跨端同步**：监听思源 `sync-end` 事件，并在插件加载后延迟合并、之后每 60 秒周期合并，自动按 ID 合并多设备数据。设置项与 API Key 采用独立时间戳合并，确保 Windows 端的设置能可靠同步到鸿蒙 NEXT 等移动端。新增设置项「跨端统计合并」开关与「立即同步」按钮——共用于一套 API Key 的多端（电脑 / 鸿蒙 / 浏览器）可显式控制合并行为，并在鸿蒙端一键拉取电脑端历史统计
- **多端适配**：支持桌面端、移动端（含鸿蒙 NEXT），UI 响应式适配
- **可视化仪表盘**：
  - 汇总卡片（总 Token / 输入 / 输出 / 请求数 / 平均耗时 / 全局限额）
  - 每个 Key 的用量进度条与阈值标记
  - Token 趋势图（支持按日/周/月聚合，自定义日期区间）：柱形按模型堆叠着色，叠加当日估算费用折线（右轴）
  - 模型用量分布
  - 最近请求记录表
  - **仪表盘自动刷新**：打开期间每 3 秒自动更新统计数据，实时反映新请求
- **费用估算**：在插件设置中按模型配置输入 / 输出单价（如 `deepseek-v4-flash` 输入 ¥0.001/千 token、输出 ¥0.004/千 token），也支持配置「资源包」——一个资源包可涵盖多个模型、包内模型共用同一单价（适合某套餐内含多个模型的场景）；仪表盘自动按模型汇总估算总费用；汇总卡片、最近请求记录表、模型用量分布均展示费用（未配置单价时显示"未配置"）。模型名匹配对大小写 / 空格 / 连字符 / 下划线不敏感，配置 `deepseek v4 flash`、`deepseek-v4-flash`、`deepseek_v4_flash` 均等效
- **单价变更自动重算（可开关）**：费用估算配置弹窗内提供「单价变更后自动重算历史费用」开关（默认开启）。开启时，调整单价后所有历史 Token 记录的费用立即按新单价实时更新（无需重算、不修改历史数据）；关闭时，每条记录在生成时冻结其费用快照，之后不再随单价变化（适合「按当时真实价格」留存费用的用户）
- **资源包额度展示**：费用估算配置的「资源包」列表中，每个资源包实时显示其涵盖模型已消耗的 token 额度——「已用 X / 总量 Y（剩余 Z）」，便于掌握按总量计费的资源包余量（资源包「总 Tokens」字段在 v1.3.19 引入，本次接入展示）
- **顶栏实时用量徽标**：顶栏图标上显示当前全局用量实时徽标——未设置任何限额时显示总 token 数（如 `1.2k`）；设置了全局 Token 限额时显示已用百分比；**设置了全局费用限额时优先以费用模式呈现**（显示已用金额百分比，悬停提示显示具体金额如 `¥22.50 / ¥50.00`）；徽标配色随阈值变化（绿 → 橙 → 红），每 3 秒自动刷新，无需打开仪表盘即可随时掌握用量；可在设置中开关「顶栏显示实时用量徽标」
- **实时阈值通知**：在原有「本端 AI 调用时触发提醒」的基础上，新增每 3 秒周期检查与云同步合并后检查，使其它设备经云同步到达本端的用量也能即时触发阈值 / 限额提醒；徽标颜色同步反映阈值状态。可在设置中开关「显示阈值提醒通知」「顶栏显示实时用量徽标」
- **全局费用限额**：在「全局 Token 限额」之外，新增独立的**费用预算上限**（金额，如 `50`，0 = 不开启）。当估算总费用达到上限时触发提醒（与 Token 阈值提醒同源机制，每 3 秒检查 + 云同步后即时触发），并可与「费用提醒阈值」百分比配合提前预警。该限额独立于按 Key 的 Token 限额，适合按月 / 按总量控制花费的场景；**已接入请求前预检**——开启「超出限额时阻止请求」后，若本次预估费用将使总额超出预算，请求会在发往服务商前被拦截（与 Token 限额三道防线对齐），而非仅提醒
- **数据管理**：仪表盘支持导出 **JSON**（完整数据备份）与 **CSV**（调用记录表 + 汇总信息，含 BOM 头便于 Excel 打开），清除记录、自动备份

## 安装

### 方式一：手动安装

1. 将 `dist/` 目录下的所有文件复制到 `思源数据目录/data/plugins/siyuan-token-stats/`
2. 在思源笔记中打开 `设置 → 集市 → 已下载 → 插件`，启用「Token 用量统计」

### 方式二：从源码构建

```bash
npm install
npm run build
```

将生成的 `dist/` 内容复制到思源插件目录。

## 使用方法

### 基本使用

1. **查看统计**：点击思源顶栏的 Token 统计图标，或使用命令面板搜索「Token 用量统计」
2. **配置 API Key**：在插件设置中点击「管理 API Key」
   - 添加思源 AI：名称随意（如"思源智能体"），Base URL 填写 `/api/ai/` 或 `http://127.0.0.1:6806/api/ai/`
   - 添加第三方 API：输入名称、API Key、提供商名称、Base URL、限额、阈值
   - **关联模型**：当多个 Key 使用相同 Base URL 时，在"关联模型"字段填写模型名（逗号分隔），插件会按请求模型匹配到对应 Key
3. **限额自动切断**：在设置中开启「超出限额时阻止请求」和「流式响应中途超限时中断」
4. **全局费用限额**：在设置中填写「全局费用限额」（金额）、「全局费用提醒阈值 (%)」，并可单独设置「全局费用重置周期」（默认每月，独立于 Token 限额周期，支持每月 / 每日 / 自定义（天）/ 永不）；如需把第三方平台的历史花费计入预算起点，可填「全局已用费用校准」。达到预算上限时底部弹出提醒（需先在「费用估算配置」中设好模型单价，否则费用为 0）
5. **阈值提醒**：当某 Key 用量达到设定百分比时，思源底部会弹出通知
5. **仪表盘自动刷新**：仪表盘打开期间每 3 秒自动更新，无需手动点击刷新
6. **费用估算配置**：在插件设置中点击「费用估算配置」
   - 选择货币类型（¥ 人民币 / $ 美元）
   - **按模型单价**：为每个模型添加输入单价与输出单价（单位：每千 token 的费用，例如 DeepSeek V3 输入 ¥0.001、输出 ¥0.004）。模型名不区分大小写，保存时自动归一化（空格 / 连字符 / 下划线均忽略，如 `deepseek v4 flash` 与 `deepseek-v4-flash` 等效）
   - **资源包**：一个资源包可涵盖多个模型，包内模型共用同一输入 / 输出单价。例如通义千问资源包填入 `qwen-turbo, qwen-plus, qwen-max` 并设单价，包内模型计费时自动套用该包价格。费用计算优先级：**单模型单价 > 资源包**。资源包还可填写「总 Tokens」（该资源包的总额度，0 = 不限），用于记录按总量计费的资源包信息
   - **单价变更后自动重算历史费用**（默认开启）：开启时改动单价后历史记录费用立即按新单价刷新；关闭时费用在记录生成时冻结为快照，不随单价变化
   - 费用公式：`输入 token ÷ 1000 × 输入单价 + 输出 token ÷ 1000 × 输出单价`；未配置单价的模型在汇总卡片与记录表中显示"—"或"未配置"，不影响其他模型统计

### 思源智能体模型识别

思源智能体（`/api/ai/agent/chat`）的请求体中 `model` 字段是**块 ID**（如 `20260705184400-rxdfohi`），不是模型名。插件通过以下机制识别真实模型：

1. **扫描 conf.json**：插件读取思源工作空间 `conf/conf.json` 中的 AI 配置（`window.siyuan.config.ai`），5 秒缓存
2. **块 ID → 模型名**：遍历 `providers[].models[]`，通过块 ID 匹配到真实模型名（如 `deepseek-v4-flash`）
3. **Key 归属匹配**：通过 provider 中的 `apiKey`、`baseURL`、`models` 字段与用户配置的 Key 匹配，确保统计归属正确

**配置建议**：在思源设置 → 人工智能中配置好 provider 和模型后，在插件的 API Key 管理中：
- 将思源 AI 的 API Key 填入插件（与 conf.json 中 provider.apiKey 一致）
- 在"关联模型"字段填写对应的模型名（如 `sensenova-6.7-flash-lite, deepseek-v4-flash`）
- 这样即使思源默认模型未配置（设为"未指定"），插件也能正确识别和归属

### 调试日志

在插件设置中开启「启用调试日志」后，浏览器控制台（Ctrl+Shift+I）会输出：
- 拦截到的 AI 调用信息（source / model / key）
- 请求体解析详情
- SSE 流式事件解析过程
- 响应分析结果（inputTokens / outputTokens / model）
- 记录写入日志

排查 token 统计为 0 或模型识别错误时，开启调试日志可快速定位问题。

## 云同步说明

插件数据存储在 `data/storage/siyuan-token-stats/data.json`，位于思源工作空间内，随思源云同步自动在设备间传输。

- **三重存储备份**：主存储（`data/storage/`）+ 插件目录备份（`data/plugins/siyuan-token-stats/settings.json`）+ localStorage，防止插件更新或异常时数据丢失
- **S3 / WebDAV 同步**：思源内核同步工作空间时会自动包含此文件；兼容华为云 S3、AWS S3、Cloudflare R2 等所有 S3 兼容存储，切换提供商无需修改插件配置
- **多设备合并**：合并在以下时机自动触发——同步完成（`sync-end` 事件）、插件加载后延迟合并、以及之后每 60 秒的周期合并（确保移动端 / 鸿蒙 NEXT 即使错过 `sync-end` 也能收敛）：
  - 调用记录：按 ID 取并集，不丢失任何设备的记录
  - API Key 配置：按 ID 合并，冲突时取 `keysUpdatedAt` 时间戳较新的一方
  - 设置项：按 `settingsUpdatedAt` 时间戳取较新的一方（v1.3.15 修复：此前曾使用全局保存时间戳，而该时间戳会在每次 Token 记录保存时被刷新，导致本端一次普通调用就把对端刚改的设置覆盖掉，表现为「Windows 设置无法同步到鸿蒙 NEXT」。现已改为设置 / 密钥各自独立的更新时间戳，互不干扰）
- **合并后回填**：若设置面板在合并时处于打开状态，最新设置会自动回填（正在编辑的字段不会被打断）

## 注意事项

- **conf.json 扫描**：插件优先从思源运行时配置 `window.siyuan.config.ai`（即 `conf/conf.json` 的加载结果）读取 AI 配置（含 providers、models），并缓存 5 秒以减少开销，用于将块 ID 解析为真实模型名。此操作只读取内存中的配置，不修改 conf.json 文件
- **模型识别优先级**：响应中的模型名 > 请求体块 ID 解析 > 全局配置兜底。确保实际使用的模型被正确记录
- **多 Key 场景**：当使用多个 API Key 管理不同大模型时，务必填写"关联模型"字段，否则同一 URL 下的请求可能归属到错误的 Key
- **Token 估算**：无 usage 字段时为近似值，精确度取决于文本类型。思源智能体的 usage 事件包含精确 token 数
- **网络拦截**：有极轻微性能开销（异步分析，不阻塞原始请求）
- **调试日志**：默认关闭，排查问题时在设置中开启即可，修改后实时生效，无需重启插件
- **鸿蒙 NEXT**：AI 请求如果通过内核发起，拦截器仍然有效
- **云同步合并**：默认自动执行（监听 `sync-end` + 加载后延迟 + 每 60 秒周期合并）；设置中提供「跨端统计合并」开关可关闭自动合并，并提供「立即同步」按钮随时随地手动拉取其他端数据

## 工作原理

### Token 拦截与识别（Interceptor）

插件通过 monkey-patch `window.fetch` 拦截思源笔记内所有 AI 调用，按以下流程统计：

1. **请求识别**：判断目标是否为 AI 接口——按 URL 模式匹配以下路径：
   - **思源智能体**（`/api/ai/agent/chat`）：思源内置的 AI 对话接口，实际由思源将请求代理转发至用户配置的第三方 API 服务商（即思源智能体使用的是第三方 API Key，非思源自有 AI）
   - **OpenAI 兼容**（`/v1/chat/completions`、`/v1/completions`）：第三方 OpenAI 格式 API
   - **Anthropic**（`/v1/messages`）：第三方 Anthropic Claude API
   - 非 AI 请求直接放行
2. **请求前预检**：从请求体 `messages` 估算输入 token（`TokenCounter.estimateFromMessages`），若开启「超出限额时阻止请求」且预估将超限，则拦截本次请求（返回 HTTP 429，不真正发往服务商）
3. **响应分析**：
   - **非流式 JSON 响应**：读取 `usage` 精确值；无 `usage` 时回退估算
   - **SSE 流式响应**（`parseSSEStream`）：逐块解析，捕获 `usage` 事件或累积 `content` 文本估算输出；生成过程中若超限且开启「流式响应中途超限时中断」，调用 `reader.cancel()` 立即中断
   - **NDJSON 流式响应**（`parseNDJSONStream`）：与 SSE 类似
4. **模型识别**：思源智能体请求体 `model` 为块 ID（如 `20260705184400-rxdfohi`），通过扫描 `conf/conf.json` 解析为真实模型名；优先级为「响应中的 model > 请求体 model > 全局配置兜底」
5. **去重写入**：`addRecord()` 写入前检查最近 5 条记录避免重复统计

### Token 提取：精确值 vs 启发式估算

| 场景 | 取值来源 | 精度 |
|------|----------|------|
| API 响应含 `usage`（OpenAI/Anthropic 格式 `usage.prompt_tokens` / `completion_tokens`，或思源智能体 `promptTokens` / `completionTokens` 事件） | 直接采用服务商返回的真实 token 数 | **精确，与官方后台一致** |
| 响应无 `usage` 字段 | 回退内置启发式估算器 `TokenCounter` | **估算，存在误差** |

`TokenCounter` 启发式（基于 GPT tokenizer 经验值，**非真实分词器**）：

```
中文汉字      ≈ 1.0 token/字
CJK 标点      ≈ 0.5 token/字
英文单词      ≈ 1.3 token/词
符号/数字/空白 ≈ 0.25 token/字符
```

公式为 `ceil(汉字×1.0 + CJK标点×0.5 + 英文词×1.3 + 其他字符×0.25)`，并叠加每条消息约 4 token、每个 `role` 约 1 token 的开销。

### 费用计算逻辑

费用在仪表盘渲染时实时计算（不写入记录），公式如下：

```
费用 = (输入 token ÷ 1000) × 输入单价 + (输出 token ÷ 1000) × 输出单价
```

- **单价来源**：用户在「费用估算配置」中按模型或资源包设置的每千 token 单价
- **匹配优先级**：单模型单价 > 资源包（包内模型共用包单价）
- **单位**：与设置的货币类型一致（¥ / $），显示保留 4 位小数（`toFixed(4)`）
- **未配置单价**：该模型费用为 0，汇总卡片 / 记录表 / 模型分布显示"—"或"未配置"
- **历史费用重算**：由「单价变更后自动重算历史费用」开关控制——开启（默认）时按当前单价实时计算；关闭时采用记录生成时写入的 `record.cost` 快照
- **费用限额**：`getTotalCostInPeriod()` 在当前重置周期内累加各记录费用，与「全局费用限额」比较触发提醒

## 统计精度与免责声明

### 精度说明

- **精确场景**：当 API 服务商在响应中返回 `usage` 字段时（主流 OpenAI 兼容接口、思源智能体 usage 事件均如此），token 数与官方后台账单一致，费用亦随之精确
- **估算场景**：若响应不含 `usage`，插件使用通用启发式估算 token 数。该估算**未调用真实分词器（如 tiktoken / cl100k_base）**，也未针对具体模型优化；对中英文混合文本误差通常约 ±20%~40%，对代码 / 符号密集文本，或采用非 GPT 系分词器的模型（如 DeepSeek、通义千问自研分词器）偏差可能更大
- **费用本身即估算**：费用 = 估算/真实的 token × 用户自填单价，单价若填错、或 token 为估算值，费用同步为估算
- **拦截覆盖**：极少数不经 `window.fetch` 发起的请求（如部分端内核直连）可能不被统计

### 免责声明

> ⚠️ **本插件所有 Token 统计与费用均为「估算 / 参考」性质，其诞生的初衷即是帮助用户在本地快速了解 AI 调用的大致消耗。请以各 API 服务商后台账单为准。**
>
> 因统计误差、部分请求未被拦截、单价配置错误、云同步合并偏差、或任何其它原因导致的统计 / 费用与实际账单的差异，本插件及开发者**不承担任何责任**。请勿将本插件数据用于精确计费、报销或合同等对准确性有严格要求的场景。
